# synthea-pipeline

> A data ingestion pipeline for **Synthea-generated synthetic healthcare events** built on top of [Spool](https://github.com/spool-framework), a reactive data pipeline framework for Java.

---

## What does this project do?

This example demonstrates how to use Spool to build a complete pipeline that:

1. **Polls every 60 seconds** a local HTTP API (`HospitalAPI`) that serves synthetic clinical events generated by [Synthea](https://github.com/synthetichealth/synthea)
2. **Normalizes** the JSON array of events into individual records, preserving the `_type` attribute to distinguish clinical event categories
3. **Persists** events to a local data lake via the ingester module
4. **Cleans up** expired events automatically through the janitor module

This pipeline is the consumer half of a two-component system — the [`HospitalAPI`](../HospitalAPI) service is the producer.

---

## Pipeline architecture

The entire pipeline is declared in a single YAML file (`pipeline.yaml`):

```yaml
modules:
  - crawler:        # Polls the HospitalAPI on a schedule
      type: POLL
      source:
        type: HTTP  # Standard Spool HTTP poll source

  - ingester:       # Persists events to the data lake
      type: REACTIVE

  - janitor:        # Removes events that exceed the configured TTL
```

```
┌──────────────────────────────────────────────────────────┐
│                    Spool Runtime                         │
│                                                          │
│  ┌──────────┐    ┌───────────┐    ┌───────────────┐     │
│  │ Crawler  │───▶│ Event Bus │───▶│   Ingester    │     │
│  │  (POLL)  │    │(IN_MEMORY)│    │  (REACTIVE)   │     │
│  └──────────┘    └───────────┘    └───────┬───────┘     │
│       │                                   │             │
│  HospitalAPI                       Data Lake (FS)       │
│  (HTTP, 20 events/req)                                   │
│                          ┌────────────┐                 │
│                          │   Janitor  │                 │
│                          │   (TTL)    │                 │
│                          └────────────┘                 │
└──────────────────────────────────────────────────────────┘
```

## Two-component system

| Component | Role |
|-----------|------|
| [`HospitalAPI`](../HospitalAPI) | FastAPI service — ingests Synthea CSV output, buffers events, and serves them via HTTP |
| `synthea-pipeline` | Spool pipeline — polls HospitalAPI, persists events to the data lake |

---

## Pipeline configuration

| Parameter | Value |
|-----------|-------|
| API URL | `http://plytrox.com:8000/events?limit=20` |
| Poll interval | 60 000 ms (1 min) |
| Root path | `events` |
| Media type | `JSON_ARRAY` |
| Tracked attribute | `_type` (clinical event category) |
| Janitor TTL | 30 000 ms |

The `_type` attribute carries the clinical event category (e.g. `encounters`, `observations`, `conditions`) and is preserved on every persisted record.

## Running the pipeline

### 1. Start the HospitalAPI

Follow the instructions in [`HospitalAPI/README.md`](../HospitalAPI/README.md) to start the companion service.

### 2. Build

```bash
mvn package
```

### 3. Run

```bash
java -jar target/synthea-pipeline.jar
```

The pipeline starts, polls the API every 60 seconds for up to 20 events, and persists them to the configured data lake partitioned by event type.

### 4. Configure paths

Edit `src/main/resources/pipeline.yaml` to change the API URL, inbox, or data lake locations:

```yaml
infrastructure:
  inbox:
    type: FILE_SYSTEM
    configuration:
      path: "/your/inbox/path"

  dataLake:
    type: FILE_SYSTEM
    configuration:
      path: "/your/datalake/path"
```

## Observability

The application exports traces, metrics, and logs via **OpenTelemetry**. By default it targets a local collector:

| Signal  | Endpoint                               |
|---------|----------------------------------------|
| Logs    | `http://localhost:3100/otlp/v1/logs`   |
| Metrics | `http://localhost:4320/v1/metrics`     |
| Traces  | `http://localhost:4318/v1/traces`      |

Compatible with any OTLP-capable backend (Grafana, Jaeger, Prometheus, etc.).
